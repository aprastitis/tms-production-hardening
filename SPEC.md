# Feature: TMS Production Hardening

Take the existing FastAPI Transaction Monitoring System from a SQLite-backed dev setup to a production-ready deploy with PostgreSQL, Alembic migrations, a real SECRET_KEY requirement, a DB health probe, a hardened multi-stage container image, CI, and a one-command compose deploy.

## Goals

- [ ] Replace SQLite with PostgreSQL via SQLAlchemy 2.x; DB URL comes from `DATABASE_URL` env var only — no SQLite fallback.
- [ ] Add Alembic with the standard layout; `env.py` reads `DATABASE_URL` from env; one initial autogenerated migration brings a fresh Postgres DB to the current schema.
- [ ] Enforce a real `SECRET_KEY` from env: app refuses to boot if unset, empty, or equal to the `changeme` placeholder. Fail-fast at settings load, before the socket binds.
- [ ] Add `GET /healthz/db`: runs `SELECT 1` against the DB. Returns 200 + `{"status": "ok", "database": "reachable"}` on success; 503 + JSON error body on failure.
- [ ] Multi-stage Dockerfile: builder stage installs deps into a venv; runtime stage copies the venv + app, runs as a non-root user, exposes the app port.
- [ ] GitHub Actions workflow on push/PR to `main`: sets up Python 3.12, installs deps, runs `pytest`, then runs `docker build` to verify the image.
- [ ] `compose.yaml` at repo root: `docker compose up` brings up Postgres 16 + the app; app waits on the Postgres healthcheck; no manual init beyond `.env`.
- [ ] Pytest smoke test exercising `/healthz/db` against a test Postgres, so CI is not vacuously green.

## Non-Goals

- TLS termination, reverse proxy, or any ingress setup (assumed external).
- AuthN/AuthZ rewrite. Only the SECRET_KEY validation changes.
- Schema redesign — same ORM models, same tables.
- Performance tuning, caching, or rate limiting.
- Kubernetes / Helm / any non-compose orchestration.
- Pushing container images from CI to a registry.
- Preserving data across the SQLite → Postgres engine swap.

## Acceptance Criteria

- [ ] `DATABASE_URL` is the only DB config source; no SQLite fallback path; no hardcoded DB paths.
- [ ] On a fresh Postgres instance, `alembic upgrade head` applies cleanly and the app starts.
- [ ] Starting the app with `SECRET_KEY` unset, empty, or `changeme` raises before binding the socket.
- [ ] `GET /healthz/db` returns 200 with the documented JSON body when the DB is reachable.
- [ ] `GET /healthz/db` returns 503 with a JSON error body when the DB is unreachable.
- [ ] `docker build .` produces an image that, when run with `DATABASE_URL` and `SECRET_KEY` set, serves requests. The Dockerfile has ≥ 2 stages and the runtime stage runs as a non-root user.
- [ ] GitHub Actions workflow at `.github/workflows/ci.yml` triggers on push/PR to `main` and runs `pytest`.
- [ ] CI workflow also runs `docker build` so a broken Dockerfile fails the pipeline.
- [ ] `docker compose up` starts both services; the app container becomes healthy (`/healthz` and `/healthz/db` both 200) with no manual DB setup beyond the `.env` file.
- [ ] Pytest suite includes at least one test that exercises `/healthz/db` against a test Postgres and asserts the documented 200 response.

## Technical Notes

- **Source of truth for the orchestrator:** no git remote. Target the local workspace path `/workspaces/transaction-monitoring-system-627deb0d/` directly. Do **not** run `git init`, do **not** create a remote, do **not** push. The orchestrator works on this path as-is.
- **Driver:** `psycopg[binary]` v3 (modern, recommended over `psycopg2-binary`).
- **ORM:** SQLAlchemy 2.x. Existing models stay; only the engine / session / config wiring changes.
- **Settings:** Pydantic v2 `BaseSettings` (or `pydantic-settings`) reading from env. `SECRET_KEY` is a required field with a validator that rejects empty / whitespace / `changeme`. `DATABASE_URL` is required.
- **Alembic layout:** standard — `alembic.ini`, `alembic/env.py`, `alembic/versions/`. `env.py` imports the app's `Base.metadata` and reads `DATABASE_URL` from env. One initial revision autogenerated from current models.
- **`/healthz/db` implementation:** FastAPI route that opens a short-lived session, executes `SELECT 1`, returns 200 or raises an HTTPException(503). No persistent DB session reused — keeps it safe under DB restarts.
- **Dockerfile:**
  - Stage 1 (`builder`): `python:3.12-slim`, create venv at `/opt/venv`, `pip install --no-cache-dir` everything from `requirements.txt`.
  - Stage 2 (`runtime`): `python:3.12-slim`, copy venv from builder, copy app code, create non-root user (`app`), `USER app`, `EXPOSE 8000`, `CMD` runs `gunicorn` with `uvicorn.workers.UvicornWorker`.
  - `.dockerignore` excludes `.git`, `__pycache__`, `.pytest_cache`, `.venv`, `*.sqlite*`, `.env`.
- **compose.yaml:**
  - `services.db`: `postgres:16-alpine`, env `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` from `.env`, named volume for data, healthcheck via `pg_isready`.
  - `services.app`: built from local Dockerfile, `depends_on.db.condition: service_healthy`, env `DATABASE_URL` and `SECRET_KEY` from `.env`, exposes `8000:8000`.
  - `.env.example` checked in (no real secrets).
- **CI (`.github/workflows/ci.yml`):**
  - Triggers: `push` and `pull_request` to `main`.
  - Job: `ubuntu-latest`, Python 3.12, pip cache, install from `requirements.txt` + `requirements-dev.txt`.
  - Services: `postgres:16` with a healthcheck; pytest uses `DATABASE_URL` pointing at it.
  - Steps: checkout → setup-python → cache pip → install deps → `pytest` → `docker build` (no push).
- **Pytest smoke test:** `tests/test_healthz_db.py` — TestClient against the FastAPI app with a session-scoped fixture pointing `DATABASE_URL` at the CI Postgres. Asserts 200 + JSON body on the happy path.
- **Pre-existing test files:** `tests/test_smoke.py` and `tests/test_flag_service.py` are in the repo but import private modules. The Developer agent should either delete or fix them as part of this scope so CI doesn't fail on import errors.
